## 1. 需求背景

### 1.1 现状与核心问题

- 角色类型当前为二元结构（职位角色/部门角色），但现状不支持“角色子级”层级能力。
- 全成本管理场景要求按角色层级进行权限限制，现状无法表达该关系，继续依赖人工多选角色组合会放大误配风险。
- 系统存在多个“特殊角色”硬编码放行菜单或全量数据，规则分散在不同模块，新增角色或调整策略需频繁改代码。
- 菜单权限虽可配置，但仍存在固定 roleId 的全放行逻辑。
- 数据权限在多个业务模块通过 `roleIds` 硬编码判断放行，缺少统一入口，模块间实现不一致，回归成本高。
- 组织信息目前主要用于主数据维护与展示回填，不是权限放行主链路输入。

### 1.2 本次目标

1. 在现有角色管理上扩展“角色子级”能力，支持全成本管理按角色层级做权限限制。
2. 菜单权限统一基于 `modules` 字段。
3. 数据权限统一基于 `dataModules` 字段。
4. 权限主键统一为 `moduleCode`，路由只消费 `modulePaths`。
5. 替换“已接入模块”的硬编码放行分支，不改模块原有隔离规则主体。
6. 采用“整体渐进迁移 + 阶段一C一次性切换（仅菜单主键切换）”的组合策略（见 [5.1](#sec-5-1)）。

### 1.3 模块归属说明（为何不在组织管理中实现权限子级）

- 组织管理职责是组织树与用户组织归属（`user.org_id`）维护。
- 当前权限链路（菜单放行、数据放行、特权分支）输入是角色，不是组织节点。
- 本次“角色及子级 + 权限基础数据”归属角色权限模块，更符合现有 RBAC 事实模型。
- 组织树与权限树逻辑独立；权限树可复用组织树的“树形基础字段形态”，但不共享权限语义（见 [2.6.4](#sec-2-6-4) 与 [4.1.2](#sec-4-1-2)）。

## 2. 统一定义（唯一事实源）

<a id="sec-2-1"></a>

### 2.1 核心术语与字段语义

- `modules`：菜单权限字段，存储可访问菜单模块的 `moduleCode` 集合（逗号分隔）。
- `dataModules`：数据权限字段，存储可触发“模块全量放行”的 `moduleCode` 集合（逗号分隔）。
- `scopeRule`：细粒度规则预留字段；本期只存不算，不参与放行判定。
- `moduleCode`：权限稳定主键（角色存储使用）。
- `modulePath`：菜单入口路由路径（导航/路由映射使用，不作为角色长期存储值）。
- `modulePaths`：鉴权接口运行时返回的 path 串（由后端 `modules(code)` 展开生成，前端直接消费）。
- `shouldBypassIsolation(userId, moduleCode)`：已接入模块的数据放行判定函数。

角色类型存量值约定：

- `type = 1`：部门角色。
- `type = 0` 或其他值（含 `null/undefined`）：职位角色。

<a id="sec-2-2"></a>

### 2.2 能力模型

#### 2.2.1 菜单权限（单层 modules）

- `modules` 只存 `moduleCode`。
- 菜单主键切换采用“阶段一C上线窗口一次性切换”策略（见 [5.4.3](#sec-5-4-3)）。

#### 2.2.2 数据权限（单层 dataModules）

- `dataModules` 只表达“该模块是否允许全量放行”。
- 未命中 `dataModules` 时，回落模块原有隔离逻辑。
- 本期不引入细粒度 ABAC 规则，`scopeRule` 预留。

#### 2.2.3 角色层级（不限制类型）

- 保留现有角色类型，新增角色父子关系字段。
- 不对 `type` 做子级能力限制，任意角色类型均可参与树关系管理。

<a id="sec-2-3"></a>

### 2.3 生效规则（判定链路唯一版本）

#### 2.3.1 菜单生效规则

1. 角色 `modules` 命中菜单对应 `moduleCode`。
2. 后端将 `modules(code)` 展开为 `modulePaths(path)` 并返回。
3. 前端基于 `modulePaths` + 既有 `menuParent` 继承进行菜单与页面判定。

#### 2.3.2 数据生效规则

1. `splitByComma(dataModules)` 精确命中 `moduleCode`：该模块允许查询全量数据。
2. 未命中：执行该模块原有数据隔离规则。
3. 未接入模块：继续走当前既有硬编码权限判断与隔离规则。

#### 2.3.3 已接入模块替换边界（必须遵守）

- 替换目标：仅替换“历史硬编码特权角色放行分支”。
- 保持不变：模块原有隔离主体（SQL 过滤、关联表过滤、负责人过滤等）。

替换前（示意）：

```text
if (命中特权角色硬编码) {
  查询全量数据
} else {
  执行模块原有数据隔离规则
}
```

替换后（示意）：

```text
if (shouldBypassIsolation(userId, moduleCode) == true) {
  查询全量数据
} else {
  执行模块原有数据隔离规则
}
```

#### 2.3.4 判定约束

- 必须精确匹配 `moduleCode`，禁止 `contains` 子串匹配。
- `false` 仅表示“不触发新放开规则”，不是“拒绝访问”；后续仍要执行模块原隔离逻辑。

<a id="sec-2-4"></a>

### 2.4 单层原则（必须满足）

- 菜单权限仅存 `modules`，数据权限仅存 `dataModules`。
- 两者独立维护、互不覆盖。
- 两者都以同一份后端模块字典（`moduleCode`）为唯一来源。

<a id="sec-2-5"></a>

### 2.5 多角色合并规则

- 菜单权限：并集合并 `modules`，命中即放行。
- 数据权限：并集合并 `dataModules`，命中即触发该模块全量放行。
- 未命中 `dataModules`：执行模块原有隔离逻辑。

<a id="sec-2-6"></a>

### 2.6 moduleCode 设计

<a id="sec-2-6-1"></a>

#### 2.6.1 设计目标

- 菜单权限与数据权限统一使用稳定 `moduleCode`。
- 路由调整时，只改字典映射，不回写角色权限数据。
- 以后端字典为唯一事实源，避免前后端编码漂移。

<a id="sec-2-6-2"></a>

#### 2.6.2 字段定义

- `moduleCode`：稳定权限编码。
  - 用途：写入 `modules`、写入 `dataModules`、作为 `shouldBypassIsolation` 输入。
  - 要求：稳定、可读、可长期复用；禁止中文、path、页面标题作为编码。
- `modulePath`：菜单主入口路由 path。
  - 用途：导航与路由映射。
  - 边界：不作为角色长期存储值。
- `modulePaths`：鉴权返回 path 串。
  - 用途：前端路由判定直接消费。
  - 生成：后端按 `modules(moduleCode)` 查字典 `module_path` 去重拼接。

<a id="sec-2-6-3"></a>

#### 2.6.3 字段关系

- 角色侧存 `moduleCode`，路由侧用 `modulePath`，通过模块字典关联。
- 默认一对一：一个 `moduleCode` 对应一个 `modulePath`。
- 若未来一对多入口，只扩展映射，不改变 `moduleCode` 主键语义。

<a id="sec-2-6-4"></a>

#### 2.6.4 字典结构与存储规则

- 字典采用树形结构：`id/name/parentId/level/idPath/isUsed/children`。
- 模块专属字段：`moduleCode/modulePath/sort`。
- 存储采用“叶子展开快照”：
  - 父节点仅做展示分组，不作为最终权限存储值。
  - 提交时展开为叶子 `moduleCode`，去重后写入 `modules/dataModules`。
- 该规则可避免“新增子节点自动继承历史权限”的隐式越权。

## 3. 前端详细设计

> 本章只描述实现落点。字段语义与判定链路统一引用 [2.1](#sec-2-1)、[2.3](#sec-2-3)、[2.6](#sec-2-6)。

<a id="sec-3-1"></a>

### 3.1 类型定义设计

在 `src/types/system.ts` 与 `src/types/auth.ts` 扩展：

```ts
type Role = {
  id: number
  role: string
  type?: RoleType
  modules: string
  parentRoleId?: number
  roleLevel?: number
  subTypeCode?: string
  dataModules?: string
  scopeRule?: string
}

type UserInfo = {
  // 现有字段...
  modules?: string
  dataModules?: string
  modulePaths?: string
}
```

说明：字段命名可按后端实际调整，但语义必须与 [2.1](#sec-2-1) 保持一致。

<a id="sec-3-2"></a>

### 3.2 角色管理页面（RoleManagement）

#### 3.2.1 页面结构

1. 角色基础信息区：角色名称、角色类型。
2. 菜单权限区：TreeSelect（按 `moduleCode` 维护 `modules`）。
3. 数据权限区：TreeSelect（按 `moduleCode` 维护 `dataModules`）。
4. 角色子级区（可对任意 `type` 生效）：
   - 是否子级角色
   - 父级角色选择
   - 子级标识编码

#### 3.2.2 表单交互规则

- 菜单 TreeSelect 改动更新 `modules`。
- 数据 TreeSelect 改动更新 `dataModules`，提交时必须发送完整字符串快照。
- 菜单权限提交值只接受 `moduleCode`，不再接受 path 语义值。

#### 3.2.3 数据模块树展示

- 节点建议字段：`name/moduleCode/parentId/level`。
- 父节点用于分组展示；提交前统一展开为叶子 `moduleCode`。
- 回显时：`dataModules` 按逗号拆分后回选，半选由组件自动计算。
- 可采用“勾选树”或“模块表开关”，语义一致。

#### 3.2.4 角色子级（树结构）交互

适用前提：角色采用树结构管理，不限制层级深度。

1. 列表（查）

- 建议“左树右表单”。
- 左树默认展示全量角色；如有业务需要，可通过 `visibleRoleTypes` 做前端展示筛选。
- 树节点结构统一：`id/parentRoleId/type/role/...`，便于后续扩展。
- 支持：搜索、展开/收起、定位当前节点。

2. 新增（增）

- 入口：新增顶级、同级、子级。
- 新增子级时自动带入 `parentRoleId=当前节点id`。
- 最小字段：`role/type/parentRoleId/modules/dataModules`。
- 成功后将新节点插入树并高亮。

3. 编辑（改）

- 可编辑：名称、父节点、菜单权限、数据权限、子类型编码。
- 改父节点必须过滤：自身节点、自身后代节点。
- 改父成功后执行“摘除旧位置 + 挂载新位置”。

4. 删除（删）

- 默认策略：有子节点不可删。
- 角色下有用户绑定不可删。
- 提示需明确阻断原因，并引导先迁移子节点/用户。

5. 一致性与并发

- 每次增删改后刷新当前分支或整树，保证一致。
- 建议携带 `updateTime/version` 乐观锁。
- 若后端返回“父节点已变更/节点不存在”，前端提示并刷新。

6. 联动边界

- 树结构只表达组织语义层级，不自动继承菜单/数据权限（本期）。
- 每个节点权限独立保存，避免隐式继承越权。

<a id="sec-3-3"></a>

### 3.3 状态管理（systemStore / authStore）

#### 3.3.1 systemStore 提交逻辑

- `handleRoleSubmit` 统一提交 `modules/dataModules` 字符串模型。
- 可选字段缺失时按默认值回填后提交（默认规则见 [5.3](#sec-5-3)）。

#### 3.3.2 authStore 权限快照

- 登录后合并 `modules/dataModules`。
- 同步缓存 `modulePaths`，用于路由判定。
- 字段语义与判定链路不在本节重复定义，统一见 [2.1](#sec-2-1)、[2.3](#sec-2-3)。

#### 3.3.3 树状态建议（角色子级）

- 维护 `roleTree`（渲染） + `roleMapById`（快速索引）。
- 工具方法：
  - `buildRoleTree(list)`
  - `findDescendantIds(roleId)`
  - `reparentNode(roleId, newParentRoleId)`
- 提交成功后优先采用后端返回结果回填，避免本地推导偏差。

<a id="sec-3-4"></a>

### 3.4 路由与菜单判定（routesConfig）

- 去除固定 roleId 的系统管理员硬编码全放行。
- 菜单与页面访问统一按 [2.3.1](#sec-2-3) 执行。
- 实现落点：`getAccessResult` 继续复用 `menuParent` 继承，不引入新权限主键。

<a id="sec-3-5"></a>

### 3.5 业务模块接入策略（前端侧）

- 首阶段不要求全部模块改造页面按钮逻辑。
- 模块接入时统一调用前端工具函数：`hasModuleAllDataAccess(moduleCode)`。
- 前端不直接依赖后端 Java 类（如 `PermissionResolverService`）。
- 前端依赖后端接口返回权限快照：
  - 菜单：`modulePaths` 参与路由判定。
  - 数据：`dataModules` 命中 `moduleCode` 参与页面行为控制。

## 4. 后端详细设计

> 本章描述后端实现形态。判定语义统一引用 [2.3](#sec-2-3)。

<a id="sec-4-1"></a>

### 4.1 数据库设计

<a id="sec-4-1-1"></a>

#### 4.1.1 角色主表扩展（role_info）

字段方案：

- `modules`（现有字段，继续用于菜单模块集合）
  - 语义：`moduleCode` 逗号分隔字符串。
  - 要求：阶段一C切换窗口一次性完成历史 path -> code 转换（见 [5.4.3](#sec-5-4-3)）。
- `data_modules` text
  - 语义：数据模块放开清单，命中 `moduleCode` 即可触发该模块全量放行。
  - 示例：`"project,contract"`。
  - 空值语义：`null/""` 表示未配置模块放开。
- `scope_rule_json` text
  - 语义：细粒度规则预留。
  - 本期约束：只存不算。
- `parent_role_id` bigint null
  - 语义：父角色ID，任意类型角色均可设置。
  - 空值：`null` 表示顶级。
- `role_level` int null
  - 语义：角色层级深度。
  - 建议：顶级 `1`，子级 `父+1`。
- `sub_type_code` varchar(64) null
  - 语义：部门子级类型编码（业务语义码，非权限主键）。
  - 示例：`HR_SPECIALIST`、`FINANCE_ASSISTANT`。
- `role_path` varchar(1024) null（可选但强烈建议）
  - 语义：祖先路径快照（如 `/1/5/23/`），用于防环与后代判断。

建议 DDL（MySQL）：

```sql
ALTER TABLE role_info
  ADD COLUMN data_modules TEXT NULL COMMENT '模块放开清单(逗号分隔字符串), 如 project,contract',
  ADD COLUMN scope_rule_json TEXT NULL COMMENT '细粒度规则预留字段(本期不参与判定)',
  ADD COLUMN parent_role_id BIGINT NULL COMMENT '父角色ID(角色子级)',
  ADD COLUMN role_level INT NULL COMMENT '角色层级深度(不限制层级建议使用INT)',
  ADD COLUMN sub_type_code VARCHAR(64) NULL COMMENT '部门子级类型编码(业务语义码)',
  ADD COLUMN role_path VARCHAR(1024) NULL COMMENT '角色祖先路径快照, 例如 /1/5/23/';
```

编码存储规范：

- `modules/data_modules` 都存稳定英文业务编码，不存中文、不存路由标题。
- 推荐小写下划线或小写英文统一风格（例：`project,contract,workplace,team,income,workload,approval,supervision`）。

<a id="sec-4-1-2"></a>

#### 4.1.2 模块字典树（module_dict，前后端共用）

建议新增独立表，沿用组织树基础字段形态：

- `id`、`name`、`parent_id`、`level`、`id_path`、`is_used`。
- 模块专属字段：`module_code`、`module_path`、`sort`。

建议 DDL：

```sql
CREATE TABLE module_dict (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '节点ID',
  name VARCHAR(128) NOT NULL COMMENT '模块名称',
  module_code VARCHAR(64) NOT NULL COMMENT '模块唯一编码',
  module_path VARCHAR(255) NULL COMMENT '菜单主入口路由path, 例如 /systemManagement/userManagement',
  parent_id BIGINT NOT NULL DEFAULT 0 COMMENT '父节点ID, 根节点为0',
  level INT NOT NULL COMMENT '层级深度',
  id_path VARCHAR(512) NULL COMMENT '节点路径快照, 例如 1-3-8',
  sort INT NOT NULL DEFAULT 0 COMMENT '同级排序',
  is_used TINYINT(1) NOT NULL DEFAULT 1 COMMENT '是否可用:1可用,0停用',
  UNIQUE KEY uk_module_code (module_code)
);
```

使用规则：

- 前端 TreeSelect 从字典树接口取数据。
- 鉴权接口返回前，后端将 `modules(code)` 展开为 `modulePaths(path 串)`。
- 角色保存时按叶子展开快照写入 `modules/dataModules`。
- 后端保存校验：`modules/dataModules` 每个编码都必须存在且 `is_used=1`。

<a id="sec-4-2"></a>

### 4.2 实体与 DTO

- `Role` 实体补齐新增字段映射。
- 新增 `ModuleDict` 实体：基础树字段 + `moduleCode/modulePath/sort`；`children` 为非持久化返回字段。
- 新增 `RoleSaveDTO/RoleDetailDTO`，避免直接暴露实体造成字段耦合。
- `RoleDetailDTO` 至少返回：`modules/dataModules/scopeRule/parentRoleId/roleLevel/subTypeCode`。

<a id="sec-4-3"></a>

### 4.3 接口设计

<a id="sec-4-3-1"></a>

#### 4.3.1 角色管理接口（扩展）

- `GET /system/role/list`：返回角色基础字段 + 树组装字段（`parentRoleId/roleLevel`）。
- `GET /system/role/get?id=...`：返回完整配置（含 `modules/dataModules/scopeRule`）。
- `POST /system/role/save`：保存基础信息 + 模块集合 + 子级关系字段。
- `GET /system/role/del?id=...`：执行子节点校验 + 用户绑定校验。

当前规模下，`list + get + save + del` 已足够支持“左树右表单”页面；无需新增树专用移动接口。

<a id="sec-4-3-2"></a>

#### 4.3.2 鉴权数据来源（沿用现有接口）

- 不新增鉴权接口，沿用现有“获取用户角色信息”接口。
- 返回字段至少包含：`modules/dataModules/modulePaths`。
- `modulePaths` 必返，由后端字典映射生成。

<a id="sec-4-3-3"></a>

#### 4.3.3 模块字典树接口

- `POST /system/module_dict/tree`
- `GET /system/module_dict/get?id=...`
- `POST /system/module_dict/save`
- `GET /system/module_dict/del?id=...`

接口要求：删除包含子节点/引用校验；前端只消费树数据与 `moduleCode`，不内置模块编码常量。

<a id="sec-4-3-4"></a>

#### 4.3.4 现有调用链改造清单（避免遗漏）

后端必须调整（接口路径可保持不变）：

- `GET /system/role/list`：补齐 `parentRoleId/roleLevel/rolePath/dataModules/scopeRule`。
- `GET /system/role/get`：返回完整新增字段，避免编辑回填丢字段。
- `POST /system/role/save`：接收并校验新增字段，新增/编辑都写入。
- `GET /system/role/del`：增加子节点校验。
- `GET /system/auth/roles`：补齐 `modules/dataModules/modulePaths`（可选带 `scopeRule`）。

前端必须调整（接口路径可保持不变）：

- 扩展角色类型定义：`Role/RoleCreateRequest/RoleUpdateRequest`。
- 扩展角色状态管理与提交流程，透传新增字段。
- 角色管理页增加 `dataModules/parentRoleId` 等字段并保证完整回显。
- 登录权限快照合并逻辑增加 `dataModules/modulePaths`。
- `getAccessResult` 使用 `modulePaths + menuParent`。

边界：以上属于“角色配置能力上线”的必做项，不属于阶段三业务模块逐步接入。

<a id="sec-4-4"></a>

### 4.4 模块权限判断组件（可选封装为 PermissionResolverService）

命名可选：`PermissionResolverService` 或 `PermissionJudgeUtil`。

定位：

- 给“已接入模块”复用判断函数，不是全系统强制替换的重型中台。
- 未接入模块不改动，继续沿用现有硬编码逻辑。

最小可用方法：

- `shouldBypassIsolation(userId, moduleCode): boolean`

可选辅助：

- `canAccessMenuPath(userId, path)`

最小流程：

1. 查用户角色集合。
2. 若任一角色 `data_modules` 命中当前 `moduleCode`，返回 `true`。
3. 否则返回 `false`。

接入约束：

- `false` 仅表示“不触发放开规则”，随后执行模块原有隔离。
- 已接入模块只替换硬编码放行分支，不改原隔离主体。

接入示意：

```text
// 硬编码分支示意
if (roleIds contains "1" or "2") {
  queryAll()
} else {
  queryWithModuleIsolation()
}

// 接入后示意
if (shouldBypassIsolation(userId, moduleCode)) {
  queryAll()
} else {
  queryWithModuleIsolation()
}
```

<a id="sec-4-5"></a>

### 4.5 角色树校验与持久化

1. 新增节点校验

- 不限制 `type`，任意角色类型都可设置 `parentRoleId`。
- `parentRoleId` 非空时父节点必须存在且启用。

2. 编辑父节点校验（防环）

- 禁止 `newParentRoleId == selfId`。
- 禁止将父节点改为自身后代。
- 建议用 `role_path` 或递归查询实现后代判断。

3. 删除校验

- 有子节点默认禁止删除。
- 有用户绑定禁止删除。

4. 层级字段维护

- 改父或保存后同步更新当前节点及全部后代的 `role_level/role_path`。
- 建议同事务内完成，避免半更新。

5. 返回规范

- 树接口稳定排序（按 `sort` 或 `id`）。
- 列表与详情字段命名保持一致。

<a id="sec-4-6"></a>

### 4.6 会话一致性与可观测性

- 角色或权限配置变更后，应刷新用户权限快照。
- 建议增加 `permission_version`：
  - Session 携带版本。
  - 版本不一致强制重新拉取权限。
- 每次权限判定记录审计日志：用户ID、角色集合、模块、判定结果、命中规则、时间。

<a id="sec-5"></a>

## 5. 兼容与迁移计划

<a id="sec-5-1"></a>

### 5.1 迁移节奏澄清（避免“渐进”与“一次性”混淆）

本文迁移策略是“**整体渐进** + **局部一次性**”：

- 整体节奏：角色能力、模块字典、判定组件、业务模块接入按阶段推进，属于渐进迁移。
- 局部一次性：仅“阶段一C菜单权限主键切换（path -> code）”要求在上线窗口一次性完成。
- 两者不冲突：
  - 阶段一C是一项有明确边界的切换动作。
  - 阶段二/三/四仍是模块化渐进替换硬编码分支。

<a id="sec-5-2"></a>

### 5.2 破坏性影响评估（仅角色模块，不含业务权限对接）

结论：在按本方案实施前提下，不对既有数据与线上功能产生不可控破坏性影响。

说明：

- “不破坏”不等于“无需改造”，仍需完成接口契约和页面字段扩展。
- 通过“路径不变 + 新字段可选 + 默认值回填”保障平滑升级。
- 本结论范围仅覆盖“角色模块能力扩展”，不包含业务模块接入阶段的改造工作量。

<a id="sec-5-3"></a>

### 5.3 字段兼容逻辑（前后端交互与展示）

后端返回（Response）：

- 基础字段：`id/role/modules/type/isUsed`。
- 扩展字段：`dataModules/modulePaths/scopeRule/parentRoleId/roleLevel/subTypeCode`。
- `modules` 语义只接受 `moduleCode`。

默认值语义：

- `dataModules=""`
- `modulePaths=""`
- `parentRoleId=null`
- `scopeRule=null`

前端归一化（Normalize）：

- 缺失可选字段时按默认值回填，保证页面不报错。
- `modules` 按 `moduleCode` 处理；路由判定直接使用 `modulePaths`。
- `type` 保持原语义用于展示与筛选，不作为是否允许挂子级的限制条件。
- `parentRoleId` 缺失视为根节点。
- `dataModules` 为空或非法格式统一归一为 `""`。

前端提交（Request）：

- 提交 `modules/type/dataModules` 与层级字段。
- 未编辑可选字段可提交默认值。

后端落库（Persistence）：

- 请求缺少可选字段时按默认值落库。
- 删除/查询接口保留原路径，但实现需补齐新增字段读写与树校验。

<a id="sec-5-4"></a>

### 5.4 分阶段计划

<a id="sec-5-4-1"></a>

#### 5.4.1 阶段一A：角色配置能力上线（不改业务判定）

- 完成角色管理新增字段与页面能力。
- 完成角色接口扩展和持久化。
- `dataModules` 允许为空字符串。
- 明确 `modules` 目标语义为 `moduleCode`，并准备历史数据转换脚本与校验方案。
- 所有业务模块继续沿用当前硬编码权限检验与原隔离逻辑。

<a id="sec-5-4-2"></a>

#### 5.4.2 阶段一B：模块字典树能力上线（不改业务判定）

- 完成 `module_dict` 表、树接口（`tree/get/save/del`）与基础校验。
- 字典节点补齐 `modulePath`，支撑鉴权返回 `modulePaths`。
- 前端数据权限 TreeSelect 接入字典树。
- 角色保存按“叶子展开快照”写入 `dataModules`。

<a id="sec-5-4-3"></a>

#### 5.4.3 阶段一C：菜单权限主键切换（一次性，无迁移期）

- 角色菜单权限编辑值切到 `moduleCode`。
- 鉴权接口稳定返回 `modulePaths`，路由判定直接消费并复用 `menuParent`。
- 在上线窗口一次性完成历史 `modules(path)` 到 `modules(code)` 回填。
- 切换完成后只接受 code 语义 `modules`，不保留 path 双读分支。

<a id="sec-5-4-4"></a>

#### 5.4.4 阶段二：权限判断组件上线（不动未接入模块）

- 提供 `shouldBypassIsolation(userId, moduleCode)`。
- 默认不改已有模块，仅为后续接入提供统一能力。

<a id="sec-5-4-5"></a>

#### 5.4.5 阶段三：模块逐步接入

- 前置：阶段一B完成。
- 按模块逐一替换“硬编码放行分支 -> shouldBypassIsolation”。
- 每接入一个模块完成独立回归后再推进下一个。

<a id="sec-5-4-6"></a>

#### 5.4.6 阶段四：硬编码收敛

- 全部模块迁移完成后，清理 roleId 硬编码判定。

- 本期不强制引入复杂 ABAC 引擎。
- 本期不改动与权限无关的业务流程。

## 7. 验收清单

1. 定义一致性

- `modules/dataModules/moduleCode/modulePath/modulePaths` 语义只在 [2.1](#sec-2-1) 与 [2.6](#sec-2-6) 定义，其他章节仅引用。
- 判定链路只以 [2.3](#sec-2-3) 为准。

2. 前后端契约

- 角色接口与鉴权接口返回字段满足 [4.3](#sec-4-3) 与 [5.3](#sec-5-3)。
- 前端类型、提交与回显满足 [3.1](#sec-3-1)、[3.2](#sec-3-2)、[3.3](#sec-3-3)。

3. 迁移边界

- 明确“整体渐进 + 阶段一C一次性”的组合策略（见 [5.1](#sec-5-1)）。
- 阶段三前，未接入模块行为不变；阶段三中仅替换硬编码放行分支。

4. 安全约束

- 模块命中必须精确匹配，不使用子串匹配。
- 角色树防环、删除校验、绑定校验生效。

<a id="sec-8"></a>

## 8. 结论

本方案以“单层 modules + 清晰判定链路 + 角色子级 + 分阶段迁移”为核心，满足：

1. 菜单与数据权限分别由 `modules/dataModules` 驱动，并统一到 `moduleCode` 主键。
2. 支持任意角色类型向子级角色扩展，且保证树结构校验可控。
3. 在不降低表达精度前提下，通过“唯一定义 + 章节引用 + 锚点跳转”降低文档重复与歧义。
4. 明确“阶段一C一次性切换”与“全局渐进迁移”并行成立，避免实施认知混淆。
