# 注释详细程度保留规范

## 何时必须用

满足以下任一场景时，**必须先参考本 skill** 再写或再改注释：

1. 任意开发任务中会新增、修订、删除注释或 JSDoc
2. 任意开发任务中会新增或修改 `type`、`interface`、DTO、共享契约、Store state、Store action、工具函数、类方法、权限/路由上下文等维护者需要先理解语义的结构
3. 修改已有注释较多、较详细的文件，尤其是工具函数、权限、加密、请求封装、并发、日志、数据库链路等约束型逻辑
4. 为统一注释风格而整理旧注释，但不确定哪些信息是“冗余”还是“有价值上下文”
5. 需要同时满足“类型字段同行注释 / 简单方法 JSDoc / 复杂方法 JSDoc + 分段解释 / 复杂 state-action 语义说明”与“保留原有详细程度”
6. 新增高约束逻辑，需要判断应该留下多细的说明，避免只写成过于简短的表面注释

典型例子：`frontend/src/utils/crypto.ts` 这类工具文件中，原注释已经说明了跨端协议、IV 拆解、密钥长度等背景；修订时要统一表达，但不要把这些有效信息删掉。

## 目标

让“注释详细程度”成为开发默认约束，而不是事后补救动作：在整个实现过程中，保留对维护者真正有帮助的上下文，避免把“详细且有用”的注释压缩成“更短但信息更少”的版本，也避免新增注释时写得过浅。

## 核心原则

### 原则 1：注释服务于理解成本，不服务于篇幅整齐

- 判断标准不是“这段代码长不长”，而是“新人第一次看到时，能不能直接知道职责、输入输出、关键约束和为什么这样设计”。
- 可以精简纯粹复述字面行为、没有信息增量的注释，但不能为了统一篇幅把有效上下文一起删掉。
- 只要某个声明需要读多个调用点、上下游协议或状态联动才能看懂，就说明当前注释仍然不够。

### 原则 2：先写语义与边界，再写表面行为

- 类型、字段、state、action、函数、方法的注释，优先解释业务语义、状态联动、权限口径、跨端约束、副作用影响。
- “为什么这样设计”“不这样做会出什么问题”“这个字段在流程里扮演什么角色”优先级高于“这里做了赋值/调用/过滤”。
- 修改已有注释时，优先修正术语、归拢结构、补齐边界，不要把 2~3 条有价值说明压缩成一句泛泛描述。

### 原则 3：注释粒度服从抽象度与维护风险，而不是代码形态

- 不是只有长函数才需要细注释；抽象名字但承载多段语义的 `type` / `interface`、聚合上下文、返回结果、Store action，同样要写清楚。
- 简单 getter / mapper / 常规透传可以保持轻量；一旦涉及权限、路由归属、异步编排、缓存、日志、数据库、副作用聚合等非直观逻辑，就要提高注释密度。
- 字段同行注释、短 JSDoc、分段说明都只是手段，目标始终是让维护者快速建立正确心智模型。

### 原则 4：注释与实现同步交付，新增声明默认要过“新人可读”检查

- 只要新增或修改类型定义、函数、方法、复杂 state/action，就默认评估并补齐注释，不把它当成事后治理任务。
- 写代码时已经识别出的协议约束、边界条件、跨端依赖、时序要求，要在当下写进注释，而不是留给后续补票。
- 新增逻辑与修订旧逻辑遵循同一标准：既不要失语，也不要堆砌没有信息增量的废话。

## 最低覆盖要求

- 新增或修改 `type` / `interface` / DTO / schema / 路由上下文 / 配置对象时，不要只写类型名；字段若承载业务语义、来源差异、判权口径、跨端约束，默认补同行注释。
- 新增或修改 Store 时，核心 state 要说明“这份状态代表什么”，核心 action 要说明“它负责哪段业务流程 / 会同步哪些状态 / 为什么要这样做”。
- 新增或修改函数、方法时：
  - 单纯 getter、格式转换、常规透传：可用短 JSDoc 或在调用关系清晰时不额外堆注释。
  - 只要涉及权限、路由归属、异步编排、缓存、日志、数据库、副作用聚合等非直观逻辑，就要补 JSDoc；必要时再写分段注释说明关键步骤。
- 对“上下文聚合类型”“返回结果类型”“中间派生结构”这类名字抽象、但实际承载多段语义的类型，至少让新人能从注释直接看懂每个字段在流程里的职责，而不是只能靠追调用猜。
- 注释目标是降低首次阅读成本，不是追求每个声明都有字数相近的说明；简单代码避免废话，复杂代码不要失语。

## 执行检查单

1. 在动代码前先看当前文件已有注释，区分“噪音”与“有效上下文”。
2. 如果原注释已经解释了协议、边界、原因、风险，默认保留，只做修订、归位和术语统一。
3. 新增或修改类型、函数、方法、复杂 state/action 时，先回答“新人第一次看到时，需要知道哪些职责、输入输出、关键约束、状态联动”。
4. 优先补业务语义、边界条件、权限口径、跨端约束、副作用影响；不要只写表面动作。
5. 类型字段优先写同行注释；方法按复杂度选择短 JSDoc 或 JSDoc + 分段说明；复杂 state/action 需要说明业务语义与状态联动关系。
6. 调整注释后，自查是否丢失了原文件中的关键信息点，以及是否仍存在需要靠追调用才能看懂的声明。
7. 若是批量治理，优先统一术语和结构，不追求所有文件注释密度完全一致。

## 反模式

- 不要把原本较详细的业务/协议注释删成一句“执行加密”“执行解密”之类的表面描述。
- 不要为了统一风格，把不同风险级别的代码写成同一种注释密度。
- 不要把“保留详细程度”误解为保留过时信息；过时内容要修正，不是原样复制。
- 不要新增大量没有信息增量的注释来凑“详细”。
- 不要把新增的 `type`、Store action、聚合上下文结构当成“名字够清楚就不需要注释”；如果字段或职责需要读多个调用点才能看懂，就说明注释还没写到位。
