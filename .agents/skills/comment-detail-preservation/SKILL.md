# 注释详细程度保留规范

## 何时必须用

满足以下任一场景时，**必须先参考本 skill** 再写或再改注释：

1. 任意开发任务中会新增、修订、删除注释或 JSDoc
2. 修改已有注释较多、较详细的文件，尤其是工具函数、权限、加密、请求封装、并发、日志、数据库链路等约束型逻辑
3. 为统一注释风格而整理旧注释，但不确定哪些信息是“冗余”还是“有价值上下文”
4. 需要同时满足“类型字段同行注释 / 简单方法 JSDoc / 复杂方法 JSDoc + 分段解释”与“保留原有详细程度”
5. 新增高约束逻辑，需要判断应该留下多细的说明，避免只写成过于简短的表面注释

典型例子：`frontend/src/utils/crypto.ts` 这类工具文件中，原注释已经说明了跨端协议、IV 拆解、密钥长度等背景；修订时要统一表达，但不要把这些有效信息删掉。

## 目标

让“注释详细程度”成为开发默认约束，而不是事后补救动作：在整个实现过程中，保留对维护者真正有帮助的上下文，避免把“详细且有用”的注释压缩成“更短但信息更少”的版本，也避免新增注释时写得过浅。

## 核心原则

### 原则 1：只删噪音，不删信息

可以删除的内容：

- 纯粹复述代码字面意思、没有额外信息的空洞注释
- 与当前实现明显不符的旧描述
- 重复三次以上且没有新增信息的同义表述

不能因为追求简洁而删除的内容：

- 跨前后端必须一致的协议约束
- 字段拼接 / 编码格式 / 常量长度的来源
- 安全、权限、并发、缓存、数据库、日志链路中的设计原因
- 对后续维护有帮助的步骤说明和阶段拆解

### 原则 2：优先“调整”，不要“降级”

修改已有注释时，优先做这些事情：

1. 修正术语、措辞和格式
2. 合并重复表述
3. 把散乱说明重组为更清晰的阶段注释
4. 把重要背景补到更准确的位置

不要做这些事情：

- 把 2~3 条有价值注释压缩成一句泛泛描述
- 只保留“做了什么”，删掉“为什么这样做”
- 因为 ESLint、审美偏好或篇幅焦虑而机械缩短注释

### 原则 3：详细程度服从代码风险，而不是篇幅平均

- 简单 getter / mapper / 常规转发逻辑：短 JSDoc 即可
- DTO / type / schema 字段：优先同行注释，直接说明字段业务语义
- 复杂方法：简短 JSDoc + 分段说明
- 高约束工具函数：允许保留更密的步骤注释，只要每条都在解释协议、边界、原因或维护风险

### 原则 4：在开发当下写到位，不要指望后补

- 写代码时如果已经识别出协议约束、边界条件、跨端依赖、时序要求，就在当下留下足够注释
- 不要先写成过度简化版本，再等后续“注释治理”时补细节
- 新增逻辑与修订旧逻辑遵循同一标准：注释密度取决于维护风险，而不是任务类型

## 执行检查单

1. 在动代码前先看当前文件已有注释，区分“噪音”与“有效上下文”。
2. 如果原注释已经解释了协议、边界、原因、风险，默认保留，只做修订与归位。
3. 新增逻辑时，判断该逻辑是否属于高约束实现；如果是，就在实现当下补足必要背景，而不是只写表面动作。
4. 调整注释后，自查是否丢失了原文件中的关键信息点。
5. 类型文件字段优先写同行注释；方法按复杂度选择短 JSDoc 或 JSDoc + 分段说明。
6. 若是批量治理，优先统一术语和结构，不追求所有文件注释密度完全一致。

## 反模式

- 不要把原本较详细的业务/协议注释删成一句“执行加密”“执行解密”之类的表面描述。
- 不要为了统一风格，把不同风险级别的代码写成同一种注释密度。
- 不要把“保留详细程度”误解为保留过时信息；过时内容要修正，不是原样复制。
- 不要新增大量没有信息增量的注释来凑“详细”。
