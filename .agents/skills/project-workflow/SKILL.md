# 项目开发工作流

## 适用场景

- 新功能开发、跨前后端联动、较大改动
- 需要先澄清输入输出、错误码、数据流和验收方式
- 任务存在多个步骤，容易在实现时失焦

完成本技能的上下文收集与计划拆解后，必须继续进入对应领域规范：前端任务转 `frontend/AGENTS.md` 与 `create-frontend-component` / `create-zustand-store`，后端任务转 `backend/AGENTS.md` 与 `create-backend-module` / `prisma-workflow` / `implement-system-log`。

## 执行步骤

### 阶段 0：上下文收集

1. 结构化扫描仓库，确认涉及 `frontend`、`backend` 还是两者联动。
2. 至少查找 3 个项目内相似实现，记录可复用文件与已有模式。
3. 明确业务目标、输入输出、异常场景、权限边界、性能/安全要求。
4. 若涉及数据库结构、核心配置、大范围重构，先标记为高风险改动。

### 阶段 1：验收契约与 TODO

1. 写清楚预期输入、输出、边界情况。
2. 列出会影响的文件 / 模块 / 页面 / 接口。
3. 明确验证方式：LSP、ESLint、TypeScript、构建、关键测试或手动验收步骤。
4. 把任务拆成可执行小步骤，并给出回滚点。

### 阶段 2：代码执行

1. 小步迭代，保持每次修改都可继续验证。
2. 优先复用现有模块结构，不自造并行体系。
3. 改实现时同步修订中文注释，尤其是业务原因、边界和异常分支。

### 阶段 3：结果验证

1. 对改动文件做诊断检查。
2. 按修改范围运行对应的 lint / typecheck / build / 测试。
3. 汇总改动文件、主要变更、验证结论、风险与后续建议。

## 禁止事项

- 不要跳过验收契约直接开改。
- 不要在没有相似实现参考的情况下随意创造新模式。
- 不要把临时文件散落到仓库任意目录，统一放 `.agent/`。
