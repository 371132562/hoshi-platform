# 多角色改造详细设计方案（一步到位版）

## 1. 背景与目标

当前系统的权限模型基于“用户单角色”设计：

- `User.roleId -> Role.id` 为单一外键关系；
- 登录态、JWT、前端菜单过滤、用户管理页均默认用户仅有一个角色；
- `Role.allowedRoutes` 是当前唯一实际生效的权限载体。

本方案目标是**一步到位**支持“用户多角色”，并明确以下原则：

1. **不保留兼容字段、不做兼容返回、不双写旧模型**；
2. 真实授权来源改为“用户-角色关联表”；
3. 登录态、JWT、前端权限消费全部切换为多角色模型；
4. 当前阶段只解决“多角色 + 菜单路由权限聚合”，不引入字段级/数据级权限。

---

## 2. 非目标

本次改造**不包含**以下内容：

- 多部门归属；
- 部门继承角色；
- 字段级、行级、审批流级权限；
- 临时授权、有效期授权、代理授权；
- 角色 deny 规则；
- 与旧单角色接口长期并存的过渡方案。

---

## 3. 核心设计原则

### 3.1 单一真相

改造完成后：

- **用户拥有的角色集合**只来源于 `UserRole`；
- `User` 表中**不再存在 `roleId` 字段**；
- 前后端都**不再消费 `user.role` 单对象**，统一改为 `user.roles`；
- 菜单权限由后端按多角色聚合后输出。

### 3.2 职责边界

- `Organization`：只表达部门归属；
- `Role`：只表达权限模板；
- `UserRole`：表达“用户被授予了哪些角色”；
- `allowedRoutes`：表达当前阶段的菜单/页面访问权限。

### 3.3 权限聚合由后端统一计算

前端不自行合并多角色权限。后端在登录、鉴权、获取用户信息时统一计算：

- `roleCodes`
- `allowedRoutes`
- `isAdmin`

前端只消费聚合结果。

## 4. 数据模型设计

## 4.1 Prisma 模型调整

### 现状

当前模型：

- `User.roleId: String`
- `User.role: Role`
- `Role.users: User[]`

### 目标模型

移除用户与角色的一对多单关系，改成多对多中间表：

```prisma
model Role {
  id            String   @id @default(uuid())
  code          String   @unique
  displayName   String
  description   String?
  isSystem      Boolean  @default(false)
  allowedRoutes Json     @default("[]")

  userRoles     UserRole[]

  createTime    DateTime @default(now())
  updateTime    DateTime @updatedAt
  delete        Int      @default(0)

  @@index([displayName, delete])
}

model User {
  id             String        @id @default(uuid())
  username       String
  displayName    String
  isSystem       Boolean       @default(false)
  phone          String?
  password       String
  organizationId String?
  organization   Organization? @relation(fields: [organizationId], references: [id])

  userRoles      UserRole[]

  createTime     DateTime @default(now())
  updateTime     DateTime @updatedAt
  delete         Int      @default(0)

  @@unique([username, delete])
  @@index([username, delete])
}

model UserRole {
  id         String   @id @default(uuid())
  userId     String
  roleId     String
  createTime DateTime @default(now())

  user User @relation(fields: [userId], references: [id])
  role Role @relation(fields: [roleId], references: [id])

  @@unique([userId, roleId])
  @@index([userId])
  @@index([roleId])
}
```

## 4.2 约束要求

### 必须满足

1. 一个用户至少拥有一个角色；
2. 同一用户不能重复绑定同一角色；
3. 软删除角色不允许继续绑定到用户；
4. 删除角色前必须确认 `UserRole` 中无有效引用；
5. 系统管理员角色仍用 `RoleCode.ADMIN` 识别。

### 建议约束

- `@@unique([userId, roleId])` 防止重复授权；
- 用户写入角色时统一先去重，再校验角色合法性。

## 5. 权限模型设计

## 5.1 多角色权限聚合规则

当前项目只存在 `allowedRoutes`，因此聚合规则明确为：

### 超管规则

只要用户角色集合中存在：

- `role.code === RoleCode.ADMIN`

则视为超管，具有全部后台菜单权限。

### 普通角色规则

对所有角色的 `allowedRoutes` 做：

1. 展平；
2. 过滤非字符串项；
3. 去重；
4. 输出并集结果。

### 当前阶段不支持

- deny 优先级；
- 权限冲突覆盖；
- 数据范围交集/并集计算。

---

## 6. 共享 DTO 设计

## 6.1 新增角色摘要类型

建议新增共享类型：

```ts
export type UserRoleRes = {
  id: string
  code?: string
  displayName?: string
  description?: string | null
  allowedRoutes?: string[]
}
```

## 6.2 用户响应类型

原有 `UserItemRes.role` 单对象模型移除，统一改成：

```ts
export type UserItemRes = {
  id: string
  username: string
  displayName: string
  isSystem: boolean
  organizationId: string | null
  organization?: { id: string; name: string } | null
  phone: string | null
  roles: UserRoleRes[]
  roleIds: string[]
  permissionSnapshot: {
    isAdmin: boolean
    allowedRoutes: string[]
  }
}
```

说明：

- `roles` 用于展示与编辑回填；
- `roleIds` 用于表单提交与简化判定；
- `permissionSnapshot` 用于前端菜单与权限判断；
- **不再返回 `role` 单对象**。

## 6.3 用户创建 / 更新 DTO

### 创建用户 DTO

```ts
export class CreateUserEncryptedReqDto {
  username: string
  displayName: string
  organizationId?: string
  phone?: string
  encryptedPassword: string
  roleIds: string[]
}
```

### 更新用户 DTO

```ts
export class UpdateUserReqDto {
  id: string
  username?: string
  displayName?: string
  organizationId?: string
  phone?: string
  roleIds?: string[]
}
```

### 列表查询 DTO

保留 `roleId?: string` 作为筛选条件，但语义调整为：

> 查询“拥有该角色”的用户。

不改成数组筛选的原因：

- 当前后台搜索交互更适合单条件筛选；
- 可减少 UI 与查询复杂度；
- 已能满足大多数按角色检索需求。

## 6.4 认证 DTO

### TokenPayloadResDto

```ts
export class TokenPayloadResDto {
  sub: string
  userId: string
  username: string
  displayName: string
  roleIds: string[]
}
```

JWT 中只保留 `roleIds`，不保留旧 `roleId`。

## 7. 后端模块详细改造设计

## 7.1 `schema.prisma`

### 变更内容

- 删除 `User.roleId` 字段；
- 删除 `User.role` relation；
- 删除 `Role.users` relation；
- 新增 `UserRole` 模型；
- `Role.userRoles` 与 `User.userRoles` 建立 relation。

### 风险

- 属于破坏性结构变更；
- 所有依赖 `include: { role: true }` 的查询都必须重写。

## 7.2 `seed.ts`

### 当前问题

系统初始化用户通过 `roleCode -> roleId` 写入 `user.roleId`。

### 新方案

初始化用户时：

1. 先创建用户本身；
2. 根据 `roleCode` 找到角色；
3. 写入 `UserRole`。

建议把 `SYSTEM_INIT_USERS` 中的 `roleCode` 扩展为 `roleCodes: string[]`，即使管理员当前只有一个角色，也统一按数组处理。

## 7.3 `user.dto.ts`

### 必改项

- `CreateUserEncryptedReqDto.roleId` → `roleIds`；
- `UpdateUserReqDto.roleId` → `roleIds`；
- `UserItemRes.role` → `roles` + `roleIds` + `permissionSnapshot`；
- `UserListReqDto.roleId` 保留，用于“包含角色”的查询。

## 7.4 `user.service.ts`

### 获取用户列表

#### 现状

- `where.roleId = roleId`
- `include: { role: true }`
- 返回 `role: { code, displayName }`

#### 新方案

- 查询改成 `include: { userRoles: { include: { role: true } }, organization: true }`；
- 如果传入 `roleId`，则使用 `userRoles.some(roleId = xxx)` 过滤；
- 返回结构改为：
  - `roles`
  - `roleIds`
  - `permissionSnapshot`

### 创建用户

流程：

1. 校验 `roleIds` 必填且非空；
2. 去重；
3. 校验所有角色存在且未删除；
4. 创建用户；
5. 批量创建 `UserRole`；
6. 所有写操作使用事务。

### 编辑用户

流程：

1. 若传 `roleIds`，先去重并校验角色；
2. 更新用户基本信息；
3. 删除该用户原有全部 `UserRole`；
4. 重建新的 `UserRole` 记录；
5. 事务提交。

## 7.5 `auth.service.ts`

### 登录返回

当前：

- 返回 `user.role`

新方案：

- 查询用户时 `include userRoles -> role`；
- 聚合 `roleIds`、`roles`、`permissionSnapshot.allowedRoutes`；
- token payload 只写 `roleIds`；
- 返回 `user.roles` + `permissionSnapshot`。

### profile 接口

与登录返回同构，确保前端刷新用户信息后字段一致。

## 7.6 `jwt.strategy.ts`

### 当前问题

请求上下文中注入的是：

- `roleId`
- `role`

### 新方案

注入：

```ts
type UserInfo = {
  userId: string
  username: string
  displayName: string
  phone?: string
  organizationId?: string | null
  organization?: { id: string; name: string } | null
  roleIds: string[]
  roles: UserRoleRes[]
  permissionSnapshot: {
    isAdmin: boolean
    allowedRoutes: string[]
  }
}
```

请求上下文只保留多角色模型，不再暴露旧单角色语义。

## 7.7 `user.decorator.ts`

同步将 `UserInfo` 类型改成多角色结构，所有控制器拿到的用户上下文统一为：

- `roleIds`
- `roles`
- `permissionSnapshot`

## 7.8 `role.service.ts`

### 删除角色校验

当前：

- `prisma.user.count({ where: { roleId: role.id }})`

新方案：

- 改为 `prisma.userRole.count({ where: { roleId: role.id }})`

### 角色用户数统计

角色列表中的 `userCount` 改为统计 `UserRole` 中关联用户数。

## 8. 前端详细改造设计

## 8.1 用户管理页 `UserManagement`

### 搜索表单

- 保持单选 `roleId` 搜索；
- 含义为“包含该角色的用户”。

### 新增/编辑表单

从：

- `name="roleId"`
- 单选 `Select`

改为：

- `name="roleIds"`
- `Select mode="multiple"`

### 列表角色展示

从单个标签改为多标签：

- 若包含 `admin`，显示红色“系统管理员”；
- 其他角色按 `displayName` 展示 `Tag`；
- 标签顺序与后端返回 `roles` 顺序一致。

## 8.2 `userStore.ts`

### 必改点

- `CreateUserFormData.roleId` → `roleIds: string[]`
- `UserPageParams.roleId` 保留（搜索筛选）
- 创建、更新用户请求改发 `roleIds`

## 8.3 `authStore.ts`

### 当前问题

本地持久化的 `user` 结构默认单角色。

### 新方案

持久化 `UserProfileResDto` 的新结构：

- `roles`
- `roleIds`
- `permissionSnapshot`

不再依赖 `user.role`。

## 8.4 `routesConfig.tsx`

### 当前问题

菜单与路由权限判断基于：

- `user.role.code`
- `user.role.allowedRoutes`

### 新方案

改为基于：

- `user.permissionSnapshot.isAdmin`
- `user.permissionSnapshot.allowedRoutes`

这样前端无需知道多角色聚合细节。

## 8.5 `UserDropdown`

### 角色展示改造

从显示单角色改为：

- 展示多个角色标签；
- 如标签过多，可限制显示前 2~3 个，其余展示 `+N`。

## 10. 数据迁移方案（一步到位）

由于本方案不做兼容，因此迁移必须一次完成。

## 10.1 迁移步骤

1. 修改 `schema.prisma`：删除 `User.roleId`，新增 `UserRole`；
2. 生成迁移；
3. 执行数据库迁移；
4. 编写迁移脚本，将历史 `User.roleId` 数据迁移到 `UserRole`；
5. 更新 seed，使系统初始化用户写入 `UserRole`；
6. 更新后端全部查询与写入逻辑；
7. 更新共享 DTO；
8. 更新前端用户管理、权限消费、登录态；
9. 完成后整体联调。

## 10.2 迁移脚本要求

迁移脚本需完成：

- 扫描所有未删除用户；
- 读取旧 `roleId`；
- 在 `UserRole` 中补入对应关系；
- 对缺失角色的异常数据进行显式告警；
- 校验每个用户至少有一个角色。

## 10.3 回滚风险

这是破坏性迁移，风险点包括：

- 一旦删除 `User.roleId`，旧版本后端无法继续读取用户角色；
- 未同步更新前端时，登录态与菜单会直接失效；
- 因此必须保证前后端同批次上线。

## 12. 验收标准

改造完成后，至少满足以下验收：

1. 用户创建 / 编辑支持多角色选择；
2. 用户列表正确展示多个角色；
3. 登录后菜单权限为多角色 `allowedRoutes` 并集；
4. 超管角色仍拥有全部权限；
5. 删除角色时若仍被任一用户引用，必须阻止删除；
6. 所有共享 DTO 与前端消费结构一致；
7. 历史单角色用户迁移后至少拥有一个 `UserRole` 记录。

---

## 13. 风险与注意事项

## 13.1 最大风险

本方案不是增量兼容，而是**整链路同步切换**。因此最大风险不是数据库迁移本身，而是：

- 后端已切多角色，但前端仍按单角色消费；
- 或前端改为 `roles`，但后端返回仍是 `role`。

这两种情况都会直接导致权限或页面异常。

## 13.2 必须遵守的实现顺序

实现时必须按以下顺序推进：

1. Prisma 模型；
2. 后端 DTO 与 service 查询；
3. 登录 / JWT / 用户上下文；
4. 前端 store；
5. 前端页面与路由权限；
6. 数据迁移与联调。

不能只改表单而不改权限链路，也不能只改 schema 而不改 DTO。

---

## 14. 最终结论

本方案是一套**一步到位的多角色改造设计**，其核心是：

- 删除 `User.roleId` 单角色真相；
- 新增 `UserRole` 作为唯一真实授权来源；
- 后端统一聚合权限；
- 前端统一消费 `roles + permissionSnapshot`；
- 不保留旧单角色兼容结构。

如果后续还要继续扩展“部门范围角色”“按钮级权限”“数据范围权限”，都应继续建立在该模型之上，而不应再回到 `User.roleId` 单字段模型。
